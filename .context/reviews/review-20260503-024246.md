## Findings

### 1. Duplicate popup-viewport-guard logic (verified)

- **Severity:** medium  
- **Category:** simplification  
- **Summary:** `keepPopupInView` is implemented twice with nearly identical logic: once as a standalone function inside `NotePopup.svelte` and again as a Svelte action inside the reader's `+page.svelte`. Both read `.hl_tip_container`, compare `dataset.adjustedTop` / `dataset.baseTop`, clamp against `window.innerHeight`, and schedule repositioning via `ResizeObserver` + timers.  
- **Evidence:**
  - `frontend/src/components/NotePopup.svelte:15–45` (`keepPopupInView` function)
  - `frontend/src/routes/library/[documentId]/+page.svelte:1350–1382` (`keepPopupInView` Svelte action, inner `update` closure)
- **Recommendation:** Extract the shared logic into a single utility (e.g., `lib/popup-viewport-guard.ts`) that accepts a `HTMLElement` and returns a cleanup function. `NotePopup.svelte` can call it imperatively from `onMount`/`onDestroy`; the reader page can wrap it in a Svelte action. This removes ~40 lines of duplication and prevents future drift.  
- **Rule candidate:** yes — "Popup positioning logic must live in a single shared utility, not be copy-pasted across components."

### 3. High cyclomatic complexity in popup repositioning closure

- **Severity:** medium  
- **Category:** simplification  
- **Summary:** The inner `update` closure inside the reader's `keepPopupInView` action scores 13 on Fallow's complexity metric. It mixes DOM reads, dataset parsing, float guards, boundary clamping, and style mutations in a single inline function.  
- **Evidence:**
  - `frontend/src/routes/library/[documentId]/+page.svelte:1350` (`const update = () => { ... }`)
  - Complexity drivers: 4+ `Number.isFinite` guards, 2 boundary `if` blocks, mutable `container.style.top` writes, and shared mutable closure state (`safeMargin`, `resizeObserver`).
- **Recommendation:** Refactor into three pure helpers:
  1. `readBaseTop(container): number | null`
  2. `clampTop(baseTop, rect, viewportHeight, safeMargin): number`
  3. `applyTop(container, top): void`
  Then `update` becomes a 4-line pipeline: `const base = readBaseTop(c); if (base == null) return; const next = clampTop(base, c.getBoundingClientRect(), ...); applyTop(c, next);`. This should drop complexity below the threshold and make the logic testable in isolation.  
- **Rule candidate:** yes — "DOM-mutation closures with >10 complexity must be decomposed into pure helper functions."

### 5. No test coverage for page-jump input or progress-bar click

- **Severity:** medium  
- **Category:** test-gap  
- **Summary:** `frontend/tests/reader-page.test.ts` covers highlight creation, note autosave, sidebar search, and colour changes, but it never exercises `handlePageJumpKeydown`, `commitPageJump`, `scrollToPage`, or `handleReadingProgressClick`. These are non-trivial interactions (Enter/Escape key handling, clamping to `[1, totalPages]`, ratio-based page jumps).  
- **Evidence:**
  - `frontend/tests/reader-page.test.ts` — grep for `pageJump`, `scrollToPage`, `reading progress`, or `numeric` returns no hits.
  - `frontend/src/routes/library/[documentId]/+page.svelte:1327–1347` (`handlePageJumpKeydown`, `commitPageJump`)
  - `frontend/src/routes/library/[documentId]/+page.svelte:1338–1347` (`handleReadingProgressClick`)
- **Recommendation:** Add two test cases:
  1. Type a page number into the `input[type="text"][inputmode="numeric"]` and press Enter — assert `scrollToPage` is called with the clamped value.
  2. Click the progress track at 50 % width — assert the reader jumps to the expected page.
- **Rule candidate:** yes — "Every user-facing input handler in the reader must have at least one component-level test."
