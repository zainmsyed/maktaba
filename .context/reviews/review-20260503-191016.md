Findings
--------

### 1) Bug — Library card delete button can be left-aligned when stats are absent
- Severity: low
- Category: bug
- Summary: When a document card has no highlight/note counts, the delete button sits at the start of the card rather than the far-right edge because the stats element (left-side) is conditionally omitted. Relying on `justify-content: space-between` alone leaves a single flex child (the button) at the start.
- Evidence: observed in user screenshot and corroborated by code in `frontend/src/routes/library/+page.svelte` (`.book-card-actions` uses `justify-content: space-between` while `.book-card-stats` is rendered conditionally). Commit cf8ba88 added `margin-left: auto` to `.delete-btn` to address this.
- Recommendation: Keep the `margin-left: auto` fix (already applied). Add a small unit/visual regression test that asserts the delete button is right-aligned regardless of `highlight_count`/`note_count` presence. Consider a short comment in the CSS noting why the auto margin is necessary (conditional sibling absence edge case).
- Rule candidate: —

### 2) Bug / UX — Notes sidebar delete alignment inconsistency
- Severity: low
- Category: bug
- Summary: The delete (✕) button in sidebar note rows can misalign when sibling content collapses. The `.paper-note-del` style relied on `.paper-note-body-btn { flex: 1 }` to push the delete button; this is fragile when markup changes or when button/div is missing.
- Evidence: sidebar markup `frontend/src/routes/library/[documentId]/+page.svelte` uses `.paper-note-body-row` without an explicit spacer. A recent edit added `margin-left: auto` to `.paper-note-del` to make the alignment robust.
- Recommendation: Keep the `margin-left: auto` on `.paper-note-del` (already applied). Add a brief visual test for the sidebar note rows (both annotation and standalone note variants) to ensure the delete button stays right-aligned. Consider documenting the intent in a small CSS comment close to the rule.
- Rule candidate: —

### 3) Dead CSS selector removed but potential duplicates remain
- Severity: info
- Category: dead-code

- Recommendation: Run a repository-wide CSS/HTML usage pass (or a grep) to find other unused classes and either remove or consolidate them. Keep the styles lean for maintainability.

### 4) Test gaps — newly-fixed behaviors lack explicit tests
- Severity: medium
- Category: test-gap
- Summary: Several recent bug fixes (reupload-restores-deleted-docs, popup-note mapping for temp IDs, and UI align fixes) were implemented but unit/integration tests for the UI alignment and the note-popup-to-backend ID mapping were not added (only a regression test for reupload exists). Visual/DOM tests are present elsewhere; these specific cases are missing.
- Evidence: Tests added in the repo include `backend/tests/test_document_upload.py` additions for reupload; frontend test files include many UI unit tests but there is no test targeting the library card delete alignment or the popup->backend highlight id translation (search for backendHighlightIdByStoreId tests yields nothing).
- Recommendation: Add the following tests:
  - Frontend unit test (Vitest + JSDOM) that renders `books-grid` with an entry having zero highlight/note counts and asserts the `.delete-btn` is at the right edge (or has `margin-left: auto`). A similar test should verify sidebar note rows keep the delete button right-aligned for empty content.
  - A frontend unit/integration test that verifies a popup note save targets the resolved backend highlight id (exercising `backendHighlightIdByStoreId`). This prevents regressions if mapping logic changes.
- Rule candidate: —

### 5) Data file accidentally tracked in git
- Severity: medium
- Category: workflow
- Summary: A PDF file (`data/pdfs/780668c60178a66e31394f81ad09256d6a9dca31b7578eac87477e9e3fcddc0b.pdf`) is listed as added in the repository working tree. Storing binary uploads in the repo can bloat git history and leak data.
- Evidence: `git status` output showed the file as staged/added.
- Recommendation: If this was accidental, remove the file from the repo and add `data/pdfs/` (or appropriate pattern) to `.gitignore`. If the file is intentionally tracked for test fixtures, move it under a `tests/fixtures` directory and document why it must be versioned. Consider using Git LFS if binary artifacts must be kept in the repo.
- Rule candidate: —

### 6) Simplification opportunity — duplicate CSS layout patterns
- Severity: info
- Category: simplification
- Summary: Multiple places implement the same layout pattern (left content, optional middle controls, delete icon right). This was solved with `margin-left: auto` per-case; a reusable utility class (e.g., `.align-action-right`) could make intent explicit and reduce duplicated fixes.
- Evidence: Both catalog cards and sidebar rows needed the same auto-margin treatment.
- Recommendation: Consider adding a small utility CSS class (documented) and apply it where appropriate instead of repeating ad-hoc margin rules. This improves discoverability and makes future maintenance easier.
- Rule candidate: - Rule candidate: Add small layout utility classes for common flex patterns.
