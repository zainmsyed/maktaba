## Findings
### Finding 1
- Severity: high
- Category: bug
- Summary: Duplicate or racing uploads can delete the canonical stored file for an already committed document.
- Evidence: In `backend/app/uploads.py:373-379`, the `except IntegrityError` handler unconditionally unlinks `final_path`. `final_path` is the deterministic hashed storage path shared by identical uploads, so if a concurrent request has already committed that file, the rollback cleanup deletes a legitimate on-disk document even though this request never created it.
- Recommendation: Track whether the current request has actually moved its temp file into `final_path` before deleting that path. On integrity failure before `os.replace`, clean up only the temp file; never delete the shared canonical file unless this request created it.
- Rule candidate: Never delete a canonical storage path during rollback unless the current transaction created that file.

### Finding 2
- Severity: medium
- Category: bug
- Summary: File validation is fail-open for structurally invalid PDFs and EPUBs.
- Evidence: `backend/app/uploads.py:116-145` only checks the PDF magic header or EPUB zip/mimetype marker, and `backend/app/uploads.py:154-174` / `backend/app/uploads.py:190-228` swallow all parser exceptions and fall back to filename metadata. A file with the right extension and superficial header can therefore be accepted even when `PdfReader` or EPUB parsing fails.
- Recommendation: Separate acceptance from metadata fallback. If the PDF/EPUB parser cannot open the file or the EPUB container/OPF cannot be read, reject the upload with 415 and delete the temp file. Keep the filename fallback only for missing metadata inside otherwise valid files.
- Rule candidate: Fail closed on upload validation; parser failures should reject the file instead of silently downgrading to filename fallback.
