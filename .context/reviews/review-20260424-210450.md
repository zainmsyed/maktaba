## Findings
### Finding 1
- Severity: medium
- Category: bug
- Summary: Ownership foreign keys do not encode the cascade behavior required by the PRD, so hard deletes will leave child rows behind instead of cleaning up pages, highlights, notes, and jobs with their parent document.
- Evidence:
  - `backend/app/models.py` defines these child references with plain `foreign_key="..."` declarations and no explicit `ondelete`: `Page.document_id`, `Highlight.document_id`, `Note.document_id`, `Note.highlight_id`, and `Job.document_id`.
  - SQLModel metadata inspection confirms the current schema intent is `ondelete=None` for those foreign keys.
  - The PRD schema in `.context/intake/prd/maktaba-prd.md` specifies `ON DELETE CASCADE` for `pages.document_id`, `highlights.document_id`, `notes.document_id`, `notes.highlight_id`, and `jobs.document_id`.
- Recommendation: Add explicit delete actions to the ownership foreign keys, matching the PRD semantics. At minimum, document-owned rows should use `ON DELETE CASCADE`, and the highlight-owned note link should also use the intended cascade behavior. Verify the generated Postgres foreign keys after the change.
- Rule candidate: When a parent row owns child rows, encode the intended `ON DELETE` action in the database foreign key instead of relying on application cleanup.

### Finding 2
- Severity: medium
- Category: workflow
- Summary: Database startup runs privileged extension DDL during normal app boot, which makes service startup depend on database-admin capabilities instead of a dedicated bootstrap or migration step.
- Evidence:
  - `backend/app/main.py` calls `initialize_database()` from the FastAPI lifespan before the app starts serving requests.
  - `backend/app/db.py` runs `connection.exec_driver_sql("CREATE EXTENSION IF NOT EXISTS vector")` unconditionally inside `initialize_database()`.
  - This couples routine application startup to extension-management privileges that many non-admin database roles do not have.
- Recommendation: Move `CREATE EXTENSION` out of normal application startup into a dedicated bootstrap/migration/admin step, or add explicit guarded handling plus operator documentation so the app can fail clearly without taking down normal boot in restricted environments.
- Rule candidate: Do not run privileged database DDL during regular application startup; keep it in migrations or explicit bootstrap flows.

### Finding 3
- Severity: low
- Category: test-gap
- Summary: Story-002 still has no automated schema smoke test, so regressions in table creation, pgvector enablement, or critical index definitions will only be caught manually.
- Evidence:
  - Repository inspection found no backend test files or pytest setup covering schema initialization.
  - The code under review adds multiple schema-critical behaviors (`SQLModel.metadata.create_all(...)`, computed FTS columns, and the ivfflat vector index) without any automated assertion that they are present after boot.
- Recommendation: Add an automated Postgres-backed smoke test that boots schema initialization and asserts the `vector` extension, required tables, and key indexes/constraints exist. Run it in CI so future schema regressions are caught automatically.
- Rule candidate: Schema/bootstrap stories should include at least one automated database smoke test for required extensions, tables, and critical indexes.
