## Findings

### Finding 1
- Severity: high
- Category: bug
- Summary: Public API responses expose internal filesystem paths (Document.file_path and job.payload.file_path), leaking server implementation details.
- Evidence:
  - backend/app/uploads.py: _job_payload includes file_path in the job payload returned to callers:

```py
def _job_payload(document: Document, step: str) -> dict[str, Any]:
    return {
        "step": step,
        "document_id": str(document.id),
        "file_path": document.file_path,
        "file_hash": document.file_hash,
        "format": document.format,
        "title": document.title,
        "authors": document.authors,
        "page_count": document.page_count,
    }
```

  - backend/app/main.py: POST /api/documents returns serialized models (document.model_dump) which include file_path:

```py
return {
    "created": result.created,
    "document": result.document.model_dump(mode="json"),
    "jobs": [job.model_dump(mode="json") for job in result.jobs],
}
```

  - backend/tests/test_document_upload.py asserts file_path is present and points to a real file (couples tests to the internal path):

```py
self.assertTrue(document["file_path"].endswith(f"{expected_hash}.pdf"))
self.assertTrue(Path(document["file_path"]).is_file())
```

- Recommendation: Do not return internal filesystem paths in public JSON payloads. Options:
  - Remove `file_path` (and other internal paths like `cover_path`) from the serialized response, leaving `file_hash`, `file_name`, and public metadata.
  - If clients need to download files, provide a stable download URL (signed or proxied) instead of exposing server filesystem layout.
  - Sanitize job.payload before returning it to external callers so it contains only client-relevant fields.
  - Update backend tests to assert the new public contract (file_hash / download_url) instead of file_path.
- Rule candidate: Do not expose internal filesystem paths or other implementation details in public API payloads.

### Finding 3
- Severity: medium
- Category: regression / bug
- Summary: GET /api/documents currently returns soft-deleted records (rows with deleted_at set), which leaks items that should be hidden from normal lists.
- Evidence:
  - backend/app/main.py uses `select(Document).order_by(...)` with no `WHERE deleted_at IS NULL` filter:

```py
docs = session.exec(select(Document).order_by(Document.updated_at.desc())).scalars().all()
```

  - models.Document defines `deleted_at` and a partial index for non-deleted rows, indicating soft-delete semantics are expected.

- Recommendation: Filter out soft-deleted records by default in list endpoints, e.g. `select(Document).where(Document.deleted_at.is_(None)).order_by(...)`. Add a test that marks a document deleted and asserts it is not returned by GET /api/documents.
- Rule candidate: List endpoints should exclude soft-deleted records by default.

### Finding 4
- Severity: medium
- Category: bug / simplification
- Summary: Frontend sorting is implemented via an opaque helper that mutates the `documents` array in-place from a reactive statement. Svelte reactivity may not reliably detect or re-run this mutation when dependencies change.
- Evidence:
  - frontend/src/routes/library/+page.svelte implements `sortDocuments()` which calls `documents.sort(...)` and is invoked via a reactive statement `$: sortDocuments();`.

```ts
function sortDocuments() {
  documents.sort((a, b) => { /* uses sortMode and document fields */ });
}

$: sortDocuments();
```

- Recommendation: Avoid in-place mutation for reactive lists. Preferred patterns:
  - Derive a sorted array reactively: `$: sortedDocuments = [...documents].sort(...)` and render `sortedDocuments`.
  - If mutating in-place, reassign `documents = documents` after mutation so Svelte sees the change (less preferred).
  - Add a lightweight UI test or E2E check that changing the sort control reorders visible items as expected.
- Rule candidate: Reactive lists should be derived from state, not mutated in place inside opaque helper calls.

### Finding 7
- Severity: low
- Category: simplification / performance
- Summary: GET /api/documents executes a separate query to fetch jobs for each document (N+1 query pattern), which will not scale for large libraries.
- Evidence:
  - backend/app/main.py loops over documents and for each runs a `select(Job).where(Job.document_id == doc.id)` query.

```py
for doc in docs:
    jobs = session.exec(
        select(Job)
        .where(Job.document_id == doc.id)
        .order_by(Job.created_at, Job.id)
    ).scalars().all()
    result.append({ ... })
```

- Recommendation: Batch-retrieve jobs for all document IDs in a single query and group them client-side, or use a JOIN/subquery to fetch related jobs more efficiently. This can be treated as a follow-up optimization (low priority) but should be addressed before the API needs to scale.
- Rule candidate: Avoid N+1 query patterns in list endpoints that fetch related rows.

### Finding 8
- Severity: medium
- Category: bug / workflow
- Summary: The library UI does not refresh job statuses after the initial load except immediately after an upload; long-running processing jobs will not transition to "ready" in the UI without a manual refresh.
- Evidence:
  - frontend/src/routes/library/+page.svelte calls `loadDocuments()` on mount and relies on replacing optimistic entries after POST, but there is no polling or subscription to update job statuses for background work.

- Recommendation: Implement a mechanism to refresh job status in the library UI:
  - Simple: poll GET /api/documents (or a dedicated job-status endpoint) every 5–15 seconds while there are processing jobs.
  - Better: use an event-driven approach (SSE / WebSocket) to push job updates to the client.
  - Add tests (integration or E2E) that assert processing -> completed transitions are visible in the UI.
- Rule candidate: —
