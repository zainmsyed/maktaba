# Review: story-004 completion review

Scope: story-004 and its direct integration points (frontend library UI + upload flow; backend listing endpoint).

Reviewer: automated code review
Date: 2026-04-25

---

## High-level summary
This review inspects the changes introduced for story-004: a new Library UI in the frontend and a GET /api/documents endpoint in the backend to support it. The core functionality is present and implements the requested UX: document listing, upload with optimistic UI, and status badges. However there are several issues and improvement opportunities that should be addressed before final closeout (see findings below). None of the findings are critical regressions blocking the feature, but a few are medium-severity and are recommended to be fixed.

---

## Findings

1) Severity: Medium
   Category: bug
   Summary: health() response contains a key string with a leading space (" data_dir")
   Evidence:
   - backend/app/main.py (excerpt):

```py
@app.get("/health")
def health() -> dict[str, object]:
    return {
        "status": "ok",
        "service": "backend",
 "data_dir": str(DATA_DIR),
        "storage_dirs": {
            "pdfs": str(DATA_DIR / "pdfs"),
            "epubs": str(DATA_DIR / "epubs"),
            "thumbs": str(DATA_DIR / "thumbs"),
        },
    }
```

   - The key string includes an extra leading space which will produce a response key of " data_dir" instead of "data_dir".

   Recommendation:
   - Remove the leading space so the returned dict key is exactly "data_dir". Example fix:

```py
"data_dir": str(DATA_DIR),
```

   - Add a small test asserting the /health route contains the expected keys (status, service, data_dir, storage_dirs).

   Rule candidate: —


2) Severity: Medium
   Category: bug / regression
   Summary: GET /api/documents returns documents without excluding soft-deleted records
   Evidence:
   - backend/app/main.py (excerpt):

```py
@app.get("/api/documents")
def list_documents(session: Session = Depends(get_session)) -> dict[str, object]:
    docs = session.exec(select(Document).order_by(Document.updated_at.desc())).scalars().all()
    for doc in docs:
        ...
```

   - models.Document includes deleted_at (soft delete field) and the application uses deleted_at elsewhere; the listing should not return records where deleted_at is set.

   Recommendation:
   - Change the query to exclude deleted documents, e.g.:

```py
from sqlalchemy import null
docs = session.exec(
    select(Document).where(Document.deleted_at == None).order_by(Document.updated_at.desc())
).scalars().all()
```

   - Add a test to verify soft-deleted documents are excluded from the listing.

   Rule candidate: API list endpoints should exclude soft-deleted records by default.


3) Severity: Medium
   Category: test-gap
   Summary: There are no automated tests for the new GET /api/documents endpoint
   Evidence:
   - Tests present: backend/tests/test_schema_smoke.py, backend/tests/test_document_upload.py
   - Neither test asserts the GET /api/documents listing behavior. test_document_upload.py covers POST uploads but not the listing.

   Recommendation:
   - Add a test (for example in backend/tests/test_documents_list.py or extending test_document_upload.py) that:
     - Uploads a PDF/EPUB via the existing POST /api/documents,
     - Calls GET /api/documents and asserts the uploaded document is listed and that associated jobs are present,
     - Asserts deleted documents are not returned (after marking/deleting one if you add a soft-delete helper in tests).

   Rule candidate: All new API endpoints should include at least one automated test.


4) Severity: Low
   Category: simplification / UX
   Summary: Frontend does not refresh job status after initial load or upload; processing badges can become stale
   Evidence:
   - frontend/src/routes/library/+page.svelte: loadDocuments() is called onMount and after an upload completes, but there is no polling or subscription to update job statuses for long-running background jobs.
   - The upload optimistic UI replaces the optimistic card when the POST completes, but background jobs (extract_text, generate_embedding) may remain in 'pending' or 'processing' for a while and the UI will not reflect their final state until manual refresh.

   Recommendation:
   - Implement a periodic refresh (poll every 5–15 seconds) for document/job status on the library view or use an event-driven approach (SSE / WebSocket) to push job updates to the client.
   - At minimum, provide a manual refresh control if live updates are undesirable for now.

   Rule candidate: Consider adopting a rule that pages showing background jobs must auto-refresh job state.


5) Severity: High
   Category: security / information-leak
   Summary: API responses may expose server filesystem paths (document.file_path and job.payload.file_path)
   Evidence:
   - backend/app/uploads.py (excerpt):

```py
def _job_payload(document: Document, step: str) -> dict[str, Any]:
    return {
        "step": step,
        "document_id": str(document.id),
        "file_path": document.file_path,
        "file_hash": document.file_hash,
        "format": document.format,
        "title": document.title,
        "authors": document.authors,
        "page_count": document.page_count,
    }
```

   - backend/app/main.py POST /api/documents returns jobs with job.model_dump(mode="json") which includes job.payload (and consequently file_path). The new GET /api/documents endpoint also returns job.model_dump(mode="json"). That means API consumers receive full file paths on the server filesystem.

   Recommendation:
   - Do not expose internal filesystem paths in API responses. Replace file_path in payload with either a safe, relative identifier (e.g. file_hash) or a generated download URL/URI if a client-facing path is required.
   - Update the job payload generation and the API serialization to omit or sanitize file_path before returning data to clients.
   - Add tests asserting that API responses do not contain absolute server-side filesystem paths.

   Rule candidate: Do not expose internal filesystem paths or other sensitive server implementation details in public API responses.

## Recommended Fixes
- [x] medium — Remove the leading space from the health() response key ("data_dir") and add a health-route key assertion test.
- [x] medium — Exclude soft-deleted documents (deleted_at IS NOT NULL) from GET /api/documents and add tests for soft-delete exclusion.
- [x] medium — Add automated tests for GET /api/documents verifying listed documents and associated jobs.
- [x] low — Implement job status refresh on the library UI (polling or SSE) so processing -> completed transitions are visible.
- [x] high — Remove or sanitize file_path from job payloads and document API responses; add tests ensuring no internal filesystem paths are exposed.
