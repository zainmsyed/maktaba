## Findings

### Finding 1
- Severity: medium
- Category: bug
- Summary: Clicking a highlight-attached sidebar note only scrolls to the page container, not to the highlight's actual source location.
- Evidence:
  - `frontend/src/routes/library/[documentId]/+page.svelte:416-425` handles sidebar note clicks by calling `scrollToPage(pageNumber)` for any note with a page number.
  - `frontend/src/routes/library/[documentId]/+page.svelte:714-723` shows `scrollToPage()` only finds `.page[data-page-number="..."]` and calls `scrollIntoView()` on the page element; it never uses the note's `highlight_id`, highlight geometry, or a highlight DOM node.
  - The current reader test only asserts that `scrollIntoView()` was called (`frontend/tests/reader-page.test.ts:320-324`), so it would still pass even if the code only jumped to the top of the page.
- Recommendation: Change the jump flow to target the specific highlight element/region rather than the page wrapper, then add a test that proves the exact highlight is focused or scrolled into view.
- Rule candidate: When a feature promises “jump back to the source location,” add an integration test against the exact annotation target, not just the page container.

---

### Finding 2
- Severity: medium
- Category: regression
- Summary: The popup editor can remain pinned while the sidebar opens a different note editor, causing two editors to render against the same shared note state.
- Evidence:
  - `frontend/src/routes/library/[documentId]/+page.svelte:335-357` and `:382-390` store note editor state in single shared variables (`activeNoteTarget`, `activeNoteRecord`, `noteDraft`, `noteStatusTone`, `noteTextareaEl`).
  - `frontend/src/routes/library/[documentId]/+page.svelte:377-379` opens a document note editor by switching `noteEditorPlacement = 'sidebar'`, but it does not clear any pinned popup state.
  - `frontend/src/routes/library/[documentId]/+page.svelte:873-887` always renders the popup editor whenever the highlighter is pinned, while `:1067-1070` independently renders the sidebar editor for document notes or sidebar highlight notes.
- Recommendation: Either scope editor state by placement/target or explicitly unpin/close the popup editor before opening a sidebar editor so only one note editor can own the shared state at a time.
- Rule candidate: When the same editor state can render in multiple placements, either scope the state per placement or explicitly close the previous placement before opening the next one.

### Finding 3
- Severity: low
- Category: bug
- Summary: Sidebar editing of a highlight note is mislabeled as a standalone document note.
- Evidence:
  - `frontend/src/routes/library/[documentId]/+page.svelte:759-804` derives the note editor heading and placeholder only from `variant === 'popup'` vs `variant === 'sidebar'`.
  - The sidebar highlight-note branch at `frontend/src/routes/library/[documentId]/+page.svelte:1069-1070` renders `noteEditor('sidebar', activeNoteTarget.highlight)`, so highlight note editing in the sidebar shows the “Document note editor” / “Standalone document note” copy even though a highlight is attached.
- Recommendation: Pass the note target kind into the editor UI and render labels/placeholders from `document` vs `highlight`, not from popup vs sidebar placement.
- Rule candidate: —

### Finding 4
- Severity: medium
- Category: test-gap
- Summary: The new note deletion path is only partially tested: there is no frontend coverage for the delete UI/client, and the backend test only covers deleting a standalone note.
- Evidence:
  - The delete client and UI exist in `frontend/src/routes/library/[documentId]/highlight-api.ts:151-210` and `frontend/src/routes/library/[documentId]/+page.svelte:427-442,1084-1103,1114-1136`.
  - `frontend/tests/reader-page.test.ts:273-357` covers note create/autosave/sidebar jump, but does not assert delete behavior.
  - `frontend/tests/highlight-api.test.ts:1-57` only covers highlight helper mapping/payload generation and does not cover the note client at all.
  - `backend/tests/test_document_upload.py:534-565` adds `test_delete_note_removes_note`, but it only covers deleting a standalone note; there is no coverage for deleting a highlight-attached note or a 404/not-found delete path.
- Recommendation: Add frontend coverage for the delete button + `deleteNote()` client path, and extend backend tests to cover deleting a highlight-attached note plus the 404 case.
- Rule candidate: When adding a destructive UI/API path, cover both the happy path and at least one failure path in client and backend tests.
