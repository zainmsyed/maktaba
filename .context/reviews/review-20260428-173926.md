## Summary
A focused review of the reader/notes visual polish changes (NoteEditor, NotePopup, reader integrations, mocks and tests). Overall the implementation is solid: a reusable NoteEditor with autosave, programmatic focus/save APIs, accessible popup wrapper, and comprehensive Vitest coverage are present. The review found a few maintenance/clarity and test gaps to address to reduce future fragility and simplify component responsibilities.

---

## Findings

1) Severity: Medium
   Category: simplification
   Summary: Autosave logic duplicated between NoteEditor and the reader page
   Evidence: NoteEditor implements a debounced autosave (500ms) and saved-badge fade (2000ms). The reader page (+page.svelte) still defines noteAutosaveTimer, noteSavedClearTimer, handleNoteInput and related autosave/status management functions and timers. This duplicates similar behaviour and increases risk of divergence when adjusting autosave or micro-UX.
   Recommendation: Consolidate autosave and saved-state timing into the NoteEditor component. Keep the page responsible only for server persistence and state updates (persistNoteDraft). Remove or mark page-level autosave timers/handlers as legacy/fallback and document expected component contract.
   Rule candidate: Encourage single-source-of-truth for autosave UI/behavior (component-level) to avoid duplicate timers and state.

2) Severity: Low
   Category: simplification
   Summary: NoteEditor exposes both callback props (onSave/onChange) and dispatches events (saved/change/error) — inconsistent component API
   Evidence: NoteEditor exports onChange/onSave props and internally dispatches events (dispatch('change'), dispatch('saved'), dispatch('error')). The reader page calls NoteEditor by passing onSave/onChange props rather than listening for events. Mixing both patterns can confuse contributors and reduces composability for third-party consumers.
   Recommendation: Standardize the public API. Prefer Svelte event semantics (on:saved, on:change) for component consumers, or document the dual API intentionally. If keeping both, consider making event emission the canonical contract and make prop callbacks a compatibility shim.
   Rule candidate: —

3) Severity: Low
   Category: test-gap (fragility)
   Summary: Test mocks use uncommon / non-standard Svelte patterns that may be brittle
   Evidence: MockPdfHighlighter.svelte uses constructs like `$props()` and `$state(...)` in the test mocks. Those patterns are atypical in regular Svelte components and can be confusing for new contributors or break if the test/build environment changes.
   Recommendation: Add a short comment in the mock source explaining why these patterns are used, or rewrite the mock using standard, explicit props and local state so intent is clearer and less environment-dependent.
   Rule candidate: —

4) Severity: Medium
   Category: test-gap
   Summary: Visual polish changes are not covered by visual / E2E tests
   Evidence: The Vitest unit tests exercise autosave, focus trap, and saved-badge fade, but there are no visual snapshot tests (screenshots) nor Playwright-style E2E checks to guard layout/regression for the new micro-UX (spacing, popup offsets, saved badge animation). Story 009 explicitly targets visual polish and spacing.
   Recommendation: Add a minimal visual regression test for the reader notes flow using Playwright (screenshot comparisons) or a lightweight screenshot test in CI for the reader route, or keep a manual QA checklist in the story and mark as required before closing. At minimum, add a local dev-check note in the story for manual verification on desktop and narrow widths.
   Rule candidate: Consider a rule: "UI polish stories should include at least one visual or E2E assertion (screenshot or Playwright) before completion." — Rule candidate: optional

5) Severity: Low
   Category: bug (potential)
   Summary: Overlapping global keydown handlers between NotePopup and NoteEditor may have order/interaction gaps
   Evidence: NotePopup registers a capturing window keydown listener (addEventListener('keydown', keydownHandler, true)), while NoteEditor also registers a non-capturing keydown listener on window to handle Escape (window.addEventListener('keydown', handleKeydown)). Different capture phases may produce ordering where both handlers run; depending on onClose side-effects this could cause double-close or unexpected focus shifts.
   Recommendation: Prefer scoping key handlers to the popup container where possible, or ensure handlers check event.defaultPrevented and short-circuit appropriately. Add a small integration test that simulates Escape when a popup editor is open to verify deterministic behavior.
   Rule candidate: —

6) Severity: Low
   Category: test-gap / contract
   Summary: NoteEditor onSave contract is implicit — onSave must return a truthy saved object (or string) otherwise it is treated as failure
   Evidence: runSave checks `if (!saved) { status = 'error'; dispatch('error', ...); return null; }` then uses saved?.content or string. If a caller accidentally returns undefined or an HTTP Response object, the editor shows an error state.
   Recommendation: Document the onSave contract (return saved note object with content property or a string) in the component comment. Optionally make runSave tolerant by treating undefined as success (fall back to draft) or by normalizing result shapes returned from onSave.
   Rule candidate: —

- [ ] Low — Add a brief comment in MockPdfHighlighter.svelte explaining the testing patterns ($props/$state) or rewrite the mock to use standard props/state to reduce fragility.
- [ ] Medium — Add at least one visual/E2E assertion (Playwright screenshot or similar) for reader notes flows that exercise popup spacing, saved-badge visibility and sidebar jump-to-highlight.
- [ ] Low — Adjust keydown handling in NotePopup/NoteEditor to avoid duplicate/conflicting handlers; add an integration test simulating Escape to verify deterministic close/focus behavior.
- [ ] Low — Document onSave return contract in NoteEditor and/or make runSave more tolerant of different return shapes from onSave.
