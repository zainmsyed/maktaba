## Findings

### Finding 1
- Severity: medium
- Category: bug
- Summary: Foreign keys that represent ownership (document → pages/highlights/notes/jobs and highlight → notes) do not declare a delete policy (ON DELETE). This risks orphaned rows when parent records are removed.
- Evidence:
  - `backend/app/models.py` defines FKs using `foreign_key="..."` but no `ondelete` argument.
  - Runtime metadata inspection showed FK ondelete values are absent:
    - pages: `document_id -> documents.id ondelete= None`
    - highlights: `document_id -> documents.id ondelete= None`
    - notes: `document_id -> documents.id ondelete= None`, `highlight_id -> highlights.id ondelete= None`
    - jobs: `document_id -> documents.id ondelete= None`
    - triples: subject/object/source_note_id ondelete=None
  - The PRD/Story expectations call out cascade behavior for document/hard-deletes and cleanup semantics for highlights → notes.
- Recommendation: Decide the intended on-delete behavior per relationship and encode it in the model foreign keys (example using SQLModel/SQLAlchemy: `Field(foreign_key="documents.id", sa_column=Column(ForeignKey("documents.id", ondelete="CASCADE")))`).
  - Typical mapping: documents → pages/highlights/jobs (ON DELETE CASCADE), highlights → notes (ON DELETE CASCADE or application decision), triples referencing notes could be `SET NULL` or `CASCADE` depending on retention requirements.
  - After model change, re-run `SQLModel.metadata.create_all(...)` in a non-destructive test environment and add a schema test to assert proper FK ondelete semantics.
- Rule candidate: When ownership semantics exist (parent → child rows), declare the FK delete action at the schema level (ondelete) rather than relying only on application cleanup.


### Finding 2
- Severity: medium
- Category: workflow / regression
- Summary: `initialize_database()` runs at FastAPI startup and executes `CREATE EXTENSION IF NOT EXISTS vector` directly. This requires superuser privileges and can fail in managed or restricted DB environments. Running administrative SQL from app startup is fragile for production deployments.
- Evidence:
  - `backend/app/db.py` contains:
    ```py
    with engine.begin() as connection:
        connection.exec_driver_sql("CREATE EXTENSION IF NOT EXISTS vector")
        SQLModel.metadata.create_all(connection)
    ```
  - `CREATE EXTENSION` typically requires a superuser; not all deployments grant that to the app DB user.
  - Starting the app in a restricted environment will raise an error at import/startup time and make boot fragile.
- Recommendation: Remove or guard superuser-level operations from automatic app startup. Options:
  1. Move extension creation to a documented admin step (run by operator with elevated privileges) and document that step in README.
  2. Wrap extension creation in a safe try/except that logs a warning when it fails and continues (so non-superuser apps still start), but provide a clear failure mode in CI or a bootstrap script that can be run by an admin.
  3. Add a `UV`/CLI script or migration step (alembic/management command) to perform admin tasks separately from normal app processes.
- Rule candidate: Do not run superuser / admin DDL (CREATE EXTENSION, cluster-wide setup) on regular application startup; perform these in a controlled admin/migration step.


### Finding 3
- Severity: low
- Category: test-gap
- Summary: There are no automated backend tests that assert the schema and critical runtime invariants (extension installed, tables present, indexes exist). Story verification was manual (compose up + introspection), which is brittle for future changes.
- Evidence:
  - Repository search for test files returned no backend tests (no pytest/ unit test files under `backend/`).
  - Current CI smoke test (present in repo) confirms services start and `GET /health` and frontend root respond, but does not assert DB schema, extensions, or index definitions.
- Recommendation: Add a small integration test (pytest) that either: (a) boots a disposable Postgres instance (or uses the compose stack) and asserts the presence of tables, the `vector` extension, and the key indexes (fts, ivfflat), or (b) runs SQL assertions against an existing test Postgres instance.
  - Hook this test into CI so schema regressions are detected automatically.
- Rule candidate: When adding schema/bootstrap logic, include at least one automated Postgres smoke test for tables, extensions, and indexes.
