## Findings
### Finding 1
- Severity: medium
- Category: dead-code
- Summary: Residual parent-side inline-note handlers and DOM refs remain after extracting the NoteEditor component and are no longer wired to the DOM.
- Evidence: frontend/src/routes/library/[documentId]/+page.svelte declares note-related fallbacks and an input handler that the template does not bind to any textarea produced by the NoteEditor component:
  - Declarations: `let noteAutosaveTimer: number | null = null`, `let noteTextareaEl: HTMLTextAreaElement | null = null` (lines near top of file)
  - Handler: `function handleNoteInput(event: Event) { ... }` schedules noteAutosaveTimer but isn't referenced by the template or passed to NoteEditor.
  - Fallback usage: openNoteEditor attempts `if (noteEditorRef?.focus) { noteEditorRef.focus(); } else { noteTextareaEl?.focus(); }` — but noteTextareaEl is never bound, so the fallback is effectively dead.
- Recommendation: Remove these leftover variables and the handler (handleNoteInput, noteTextareaEl, noteAutosaveTimer) if the NoteEditor component is the sole editor. If a plain-text fallback is intentionally kept, explicitly bind the textarea and document the reason for maintaining both approaches. Removing this dead code will reduce cognitive overhead and avoid confusion.
- Rule candidate: When extracting a UI component, remove parent-side inline handlers/refs unless an explicitly documented fallback is required.

### Finding 2
- Severity: medium
- Category: simplification
- Summary: Autosave and save-status logic is implemented in both NoteEditor.svelte and the reader page; this duplication risks inconsistent UX and subtle race conditions.
- Evidence: NoteEditor.svelte maintains its own autosave timer, runSave/save API, status states and scheduleSavedNoteFade; concurrently the reader page manages noteDraft, noteSavedDraft, noteSavedClearTimer, scheduleSavedNoteFade, setNoteStatus, and persistNoteDraft which performs backend create/update. See:
  - NoteEditor.svelte: autosaveTimer, runSave(), dispatch('saved'), scheduling of fade (500ms autosave, 2000ms fade).
  - frontend/src/routes/library/[documentId]/+page.svelte: persistNoteDraft() (calls API), scheduleSavedNoteFade(), noteAutosaveTimer and caller logic.
- Recommendation: Move autosave timers and immediate save-status UI entirely into NoteEditor (it already exposes onSave/onChange and save()); the parent should rely on the component's onSave/onSaved events and only perform high-level server merges (upsertNote) or listen to save completion. This reduces duplicate timers, simplifies reasoning, and reduces the chance of out-of-order saves between component and parent state.
- Rule candidate: Prefer single-source-of-truth for input autosave and save-status; the component owning the input should manage timers and UX and expose an async save API/events for parents.

### Finding 3
- Severity: medium
- Category: test-gap
- Summary: The create_highlight endpoint (POST /api/documents/{document_id}/highlights) exercises PDF text extraction (PyMuPDF) but lacks direct backend integration tests exercising the HTTP path.
- Evidence: backend/app/main.py implements create_highlight which opens the document with fitz, computes a clip rect, and calls page.get_text('text', clip=clip) before creating the Highlight row. Backend test suite covers uploads, worker extraction, and DB-level highlight/note interactions (backend/tests/test_document_upload.py and backend/tests/test_extract_worker.py) but does not perform an HTTP POST to /api/documents/{id}/highlights to validate the extraction/handler behaviour and error mapping.
- Recommendation: Add an integration test that uploads a simple PDF, then POSTs to /api/documents/{document_id}/highlights with valid geometry and asserts the response highlight contains expected extracted_text and persisted fields. Also add tests for failure modes (invalid page number, missing file, PyMuPDF not available if possible). This prevents regressions in the extraction endpoint and ensures consistent API messages.
- Rule candidate: Endpoints that exercise media processing (PDF text extraction, OCR) should have integration tests covering success and common failure modes.

### Finding 4
- Severity: low
- Category: bug (error-handling)
- Summary: The create_highlight handler converts arbitrary exceptions into HTTP 500 responses returning the raw exception string, which may leak internal details.
- Evidence: In backend/app/main.py:

  ```py
  except Exception as exc:
      raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail=str(exc))
  ```

  This pattern appears in other handlers in the codebase as well.
- Recommendation: Log the full exception (traceback) server-side and return a generic 5xx message to clients (e.g., "Internal server error processing PDF extraction"). Map expected validation errors and resource-not-found cases to 4xx responses where appropriate. This improves security and reduces noisy client error messages.
- Rule candidate: Do not return raw exception strings in 5xx responses; log internals and return a generic message.
