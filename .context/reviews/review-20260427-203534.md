## Findings
### Finding 1
- Severity: medium
- Category: bug
- Summary: Sidebar "jump" only scrolls to the page and does not focus or open the specific highlight element or its inline editor/popup.
- Evidence: In frontend/src/routes/library/[documentId]/+page.svelte the handler handleSidebarNoteClick(note) calls scrollToPage(pageNumber) and sets statusMessage but does not attempt to locate the highlight DOM element or call openHighlightNoteEditor when note.highlight_id is present. The code:
  - handleSidebarNoteClick(note) { const pageNumber = getNotePageNumber(note); if (pageNumber === null) { void openDocumentNoteEditor(note); return; } scrollToPage(pageNumber); statusMessage = `Jumped to page ${pageNumber}`; }
  - Test coverage in frontend/tests/reader-page.test.ts asserts that scrollIntoView is called when a sidebar entry is clicked, but does not assert that the highlight itself is focused or that the inline note editor/popup opens.
- Recommendation: For highlight-attached notes, locate the specific highlight element (by note.highlight_id) after scrolling and call scrollIntoView on that element or call openHighlightNoteEditor(...) to surface the inline editor/popup. Ensure the implementation is robust if the highlight is not present (fall back to page scroll). Add unit tests that assert the highlight-level jump behavior (e.g., spy on the element scrollIntoView or confirm the popup/editor is opened).
- Rule candidate: When a sidebar entry references inline document content (highlights), clicking it should focus or open the source element rather than only scrolling to the page.

### Finding 2
- Severity: medium
- Category: test-gap
- Summary: Autosave behaviour is covered for the happy path but there are no tests for failure modes and race conditions (network errors during create/update, cancelling autosave while switching editors, and deletion while editing).
- Evidence: frontend/tests/reader-page.test.ts covers autosave happy path and doc-note creation, but there are no tests simulating server failures (create/update returning 5xx), nor tests that cover rapid editor switching or deletion during an in-flight autosave. The code paths of persistNoteDraft, flushActiveNoteDraft, and handleNoteInput include branching for error handling (setNoteStatus('error')), but these branches are not exercised in tests.
- Recommendation: Add Vitest tests that:
  - Simulate create/update endpoints returning non-2xx and assert the UI shows the 'error' status and recovers on subsequent input.
  - Verify that switching editors triggers flushActiveNoteDraft and that the previous draft is persisted (or error state handled) before opening the new editor.
  - Verify deletion while editing closes the editor and does not leave dangling timers/requests.
  - Consider adding tests that confirm scheduleSavedNoteFade and revision checks avoid stale updates.
- Rule candidate: When implementing autosave, include tests for success, failure, cancellation and editor-switch races.

### Finding 3
- Severity: low
- Category: simplification / typing
- Summary: Type annotation for unsubscribeHighlightsStore is misleading — uses (() => boolean) | null but typical unsubscribe functions return void.
- Evidence: In frontend/src/routes/library/[documentId]/+page.svelte the variable is declared as: let unsubscribeHighlightsStore: (() => boolean) | null = null; The mock HighlightsModel.subscribe returns a cleanup function that calls Set.delete(callback) which returns a boolean; but most subscribe/unsubscribe patterns return void. This mismatch may produce confusion or TypeScript errors under stricter compiler settings.
- Recommendation: Change the type to (() => void) | null (or align the subscribe implementation to return void) to reflect the intended usage. Prefer void for unsubscribe functions to avoid accidental reliance on the boolean return value of Set.delete.
