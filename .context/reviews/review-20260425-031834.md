# Review: story-004 completion review

Scope: story-004 and its direct integration points (frontend library UI + upload flow; backend listing endpoint).
Trigger: complete-story
Date: 2026-04-25

---

## High-level summary
The library UI and upload flow have been implemented and wired to the backend. The change set is functionally complete for a minimal workflow (upload -> appear in library), but there are several correctness, security, and quality issues that should be addressed before final story closeout. The priority items are: (1) remove internal filesystem paths from public API responses, (2) exclude soft-deleted documents from the listing, (3) make frontend sorting reliably reactive, and (4) add tests covering the new listing endpoint and library behaviors. There are additional minor simplifications and a potential N+1 performance concern.

---

## Findings

### 1) Public API responses leak internal filesystem paths
- **Severity:** High
- **Category:** security / bug
- **Summary:** API responses returned by POST /api/documents and GET /api/documents currently include server-side filesystem paths (Document.file_path and job.payload.file_path). This exposes internal storage layout to API consumers.
- **Evidence:**
  - backend/app/uploads.py (job payload includes file_path):

```py
def _job_payload(document: Document, step: str) -> dict[str, Any]:
    return {
        "step": step,
        "document_id": str(document.id),
        "file_path": document.file_path,
        "file_hash": document.file_hash,
        "format": document.format,
        "title": document.title,
        "authors": document.authors,
        "page_count": document.page_count,
    }
```

  - backend/app/main.py (POST response serializes full models including file_path):

```py
return {
    "created": result.created,
    "document": result.document.model_dump(mode="json"),
    "jobs": [job.model_dump(mode="json") for job in result.jobs],
}
```

  - backend/tests/test_document_upload.py asserts file_path exists and reads the file on disk (test expects file_path to be exposed):

```py
self.assertTrue(document["file_path"].endswith(f"{expected_hash}.pdf"))
self.assertTrue(Path(document["file_path"]).is_file())
```

- **Why this matters:** Exposing absolute server paths reveals internal implementation details and can create security or portability issues. It also couples client expectations to internal storage layout (tests rely on it), making future refactors harder.
- **Recommendation:** Stop returning internal filesystem paths in public API responses. Instead:
  - Remove `file_path` from public JSON payloads (both document and job payloads) or replace with a safe, client-oriented field such as `file_hash`, `file_name`, or a generated download URL (signed or proxied) if clients need to download the file.
  - Sanitize job.payload before serializing to external consumers so it does not contain internal file paths.
  - Update backend tests to assert on the new public contract (file_hash or download URL) instead of `file_path`.
- **Rule candidate:** Do not expose internal filesystem paths or other implementation details in public API payloads.


### 2) GET /health returns a malformed JSON key (leading space)
- **Severity:** Medium
- **Category:** bug
- **Summary:** The /health JSON object contains a key with a leading space (`" data_dir"`) due to stray whitespace in the code. This will break any strict key lookups by clients.
- **Evidence:** backend/app/main.py

```py
@app.get("/health")
def health() -> dict[str, object]:
    return {
        "status": "ok",
        "service": "backend",
 "data_dir": str(DATA_DIR),
        "storage_dirs": {
            "pdfs": str(DATA_DIR / "pdfs"),
            "epubs": str(DATA_DIR / "epubs"),
            "thumbs": str(DATA_DIR / "thumbs"),
        },
    }
```

- **Recommendation:** Remove the stray leading space so the key is exactly `"data_dir"`. Add a small unit/testclient test asserting the health route contains the expected keys.
- **Rule candidate:** —


### 3) GET /api/documents returns soft-deleted documents
- **Severity:** Medium
- **Category:** bug / regression
- **Summary:** The listing endpoint currently selects all Document rows and does not exclude rows with `deleted_at` set (soft-deletes), despite the schema supporting soft-delete semantics.
- **Evidence:** backend/app/main.py

```py
docs = session.exec(select(Document).order_by(Document.updated_at.desc())).scalars().all()
```

`models.Document` defines `deleted_at` and a partial index for non-deleted rows.

- **Recommendation:** Filter out soft-deleted records in the listing query, e.g. `select(Document).where(Document.deleted_at.is_(None)).order_by(...)`. Add a test that inserts a document with `deleted_at` set (or marks it deleted) and asserts it is not returned by GET /api/documents.
- **Rule candidate:** List endpoints should exclude soft-deleted records by default.


### 4) Frontend sorting is non-reactive (in-place mutation + opaque helper)
- **Severity:** Medium
- **Category:** bug / simplification
- **Summary:** The current implementation calls a helper `sortDocuments()` from a reactive statement `$: sortDocuments();`. The helper sorts `documents` in place via `documents.sort(...)`. Svelte's compiler/ reactivity model does not track dependencies inside the helper body, so the reactive statement may not rerun when `sortMode` or `documents` change, and in-place mutation can be missed by change detection.
- **Evidence:** frontend/src/routes/library/+page.svelte

```svelte
function sortDocuments() {
  documents.sort((a, b) => { /* uses sortMode and document fields */ });
}

$: sortDocuments();
```

- **Recommendation:** Derive a new, reactive sorted array and render that. Example pattern:

```svelte
$: sortedDocuments = [...documents].sort((a, b) => {
  // sorting logic referencing sortMode
});
```

Render `sortedDocuments` in the template. Alternatively, after in-place mutation, reassign `documents = documents` (or better, replace with a new array) to ensure Svelte notices the change. Prefer the derived array approach for clarity and correctness.
- **Rule candidate:** Reactive lists should be derived from state, not mutated in place inside opaque helper calls.


### 5) Missing automated tests for GET /api/documents and library UI behaviors
- **Severity:** Medium
- **Category:** test-gap
- **Summary:** The new GET /api/documents endpoint and frontend behaviors (sorting, optimistic upload replacement) are not covered by automated tests. Existing tests exercise upload POST flows but not the listing or the UI-level interactions.
- **Evidence:** backend/tests/test_document_upload.py tests upload endpoints but never calls `GET /api/documents`. There is no backend test file for the listing endpoint and no frontend/e2e tests for the library page.
- **Recommendation:** Add tests:
  - Backend integration test: upload one or more documents, call GET /api/documents, assert the expected document(s) and associated jobs are present and that deleted documents are excluded.
  - Update existing tests that relied on `file_path` to use the new sanitized contract.
  - If feasible, add a lightweight frontend component or e2e test that verifies: optimistic upload card insertion and replacement after upload, and that the sort control changes displayed order.
- **Rule candidate:** All new API endpoints should include at least one automated test.


### 6) Unused global `uploading` state in frontend
- **Severity:** Low
- **Category:** dead-code / simplification
- **Summary:** The component defines a global `let uploading = false` flag that is set during uploads but never used in the template. The per-item optimistic upload state (`entry.uploading`) is used for UI, so the global flag is redundant and confusing.
- **Evidence:** frontend/src/routes/library/+page.svelte

```ts
let uploading = false;
...
async function upload(file: File) {
  uploading = true;
  ...
  } finally {
    uploading = false;
  }
}
```

The template does not reference `uploading`.

- **Recommendation:** Remove the unused global `uploading` variable, or intentionally use it (e.g., to disable the upload control while any upload is in progress). Prefer per-item state for concurrent uploads.
- **Rule candidate:** Avoid unused global UI state; prefer per-item state for concurrent operations.


### 7) Potential N+1 query pattern when loading jobs per document
- **Severity:** Low
- **Category:** simplification / performance
- **Summary:** The listing endpoint fetches all documents and then executes a query per document to fetch jobs, which is an N+1 query pattern that will scale poorly for larger libraries.
- **Evidence:** backend/app/main.py

```py
docs = session.exec(select(Document).order_by(Document.updated_at.desc())).scalars().all()
for doc in docs:
    jobs = session.exec(
        select(Job)
        .where(Job.document_id == doc.id)
        .order_by(Job.created_at, Job.id)
    ).scalars().all()
    result.append({ ... })
```

- **Recommendation:** Batch-retrieve jobs for all document IDs in one query and group them client-side (or use a join/subquery) to avoid N+1 queries. This is an optimization recommendation and can be deferred until the API needs to scale.
- **Rule candidate:** Avoid N+1 query patterns in list endpoints that fetch related rows.

## Recommended Fixes
- [x] high — Remove internal filesystem paths from API responses; replace with safe identifiers (file_hash/file_name) or generated download URLs and update tests to the new contract.
- [x] medium — Fix the `/health` JSON key typo (remove leading space) and add a health-route test.
- [x] medium — Exclude soft-deleted documents from `GET /api/documents` and add tests verifying deleted-row exclusion.
- [x] medium — Make frontend sorting reactive by deriving `sortedDocuments` from `documents` and `sortMode` (do not rely on in-place mutation inside an opaque helper).
- [x] medium — Add backend integration tests for `GET /api/documents` and optional frontend tests for optimistic upload replacement and sorting.
- [x] low — Remove or repurpose the unused global `uploading` state in the frontend component.
- [x] low — Optimize `GET /api/documents` to avoid N+1 queries for jobs (batch select or join) when scaling is required.
