## Findings
### Finding 1
- Severity: medium
- Category: bug
- Summary: Failed `extract_text` jobs can still commit partial page rows because the worker never rolls back staged `Page` inserts before marking the job failed.
- Evidence:
  - `backend/app/worker.py` stages page rows in the session during the success path:

```py
for page_number, text, ocr_used in extract_text_and_fallback(pdf_path):
    p = Page(
        document_id=doc.id,
        page_number=page_number,
        extracted_text=text or None,
        ocr_used=bool(ocr_used),
    )
    pages_to_add.append(p)
...
for p in pages_to_add:
    ...
    session.add(p)
```

  - In the exception handler it sets the job to failed and commits, but never calls `session.rollback()`:

```py
except Exception as exc:
    logger.exception("Failed to process extract_text job %s: %s", getattr(job, "id", None), exc)
    job.status = "failed"
    job.error_message = str(exc)
    session.add(job)
    session.commit()
```

  - If extraction or OCR fails after some pages have been staged, those pending `Page` objects are flushed by the final commit and the document ends up with partial page text even though the job failed.
- Recommendation: Roll back the session before recording the failure, or separate page insertion from job status updates so a failed extraction cannot commit partial `Page` rows. Add a failure-path test that simulates an exception mid-document and asserts no `Page` rows persist.
- Rule candidate: If a background job writes child rows, rollback staged inserts before recording failure so partial output never commits.

### Finding 2
- Severity: medium
- Category: test-gap
- Summary: The new PDF extraction/OCR worker path has no automated coverage, so the story still relies on manual verification for the text-PDF happy path, the scanned-PDF OCR fallback, and the rollback behavior above.
- Evidence:
  - `backend/tests/test_document_upload.py` covers upload/listing/streaming, but there is no test for `backend/app/extract.py` or `backend/app/worker.py`.
  - `rg "extract_text" backend/tests -n` returns no hits, so there is no backend test asserting page rows or `ocr_used` for either a text PDF or a scanned PDF.
- Recommendation: Add backend tests for `extract_text_and_fallback` / `process_extract_job` (or a worker integration harness) that cover: a text-layer PDF (`ocr_used = false`), the scanned PDF OCR fallback (`ocr_used = true`), and the failure rollback case.
- Rule candidate: New background extraction/OCR flows should have automated coverage for both the happy path and failure rollback.
