## Findings

### Finding 1
- Severity: high
- Category: workflow
- Summary: Local virtual environment (.venv) was created in-repo and could be accidentally committed.
- Evidence:
  - `find backend -maxdepth 2 -type f` returned backend/.venv artifacts:

    backend/.venv/CACHEDIR.TAG
    backend/.venv/.gitignore
    backend/.venv/.lock
    backend/.venv/pyvenv.cfg

  - A `.gitignore` was added at `backend/.gitignore` to exclude `.venv/` but the `.venv` directory exists in the working tree (risk of accidental commit before ignore takes effect).

- Recommendation: Remove any .venv files from version control if accidentally staged/committed and ensure .venv is ignored. Steps:
  1. If `.venv` was staged previously, run: `git rm -r --cached backend/.venv`
  2. Ensure `backend/.gitignore` contains `.venv/` (already present).
  3. Commit the `.gitignore` change and verify: `git status` shows no .venv files staged.

- Rule candidate: When creating a local Python virtualenv inside a repo, add it to `.gitignore` and remove it from the index immediately to avoid committing bulky artifacts.

### Finding 2
- Severity: medium
- Category: workflow / regression
- Summary: `uv.lock` was generated on the host Python (CPython 3.13.5) while the Docker build uses Python 3.12; lock generation and container Python minor version mismatch can cause install/build failures for CPython-specific wheels.
- Evidence:
  - `uv lock` output: `Using CPython 3.13.5`
  - `backend/pyproject.toml` requires-python = ">=3.12"
  - `backend/Dockerfile` bases builder and runtime on `python:3.12-slim` (`FROM python:3.12-slim AS builder`)

- Recommendation: Re-generate a lockfile aligned to the container Python version (3.12), or explicitly pin the `requires-python` to the range matching the container. For example:
  - Locally: `cd backend && uv lock --python 3.12`
  - Or set `requires-python = ">=3.12,<3.13"` in `pyproject.toml` and re-run `uv lock` to produce a 3.12-compatible `uv.lock`.

- Rule candidate: When generating lockfiles for containerized builds, generate the lock with the same Python minor version used in the containers (e.g., `uv lock --python X.Y`).

### Finding 3
- Severity: medium
- Category: bug / simplification
- Summary: Backend Docker builder stage runs `uv sync` in a `python:3.12-slim` image without installing OS-level build dependencies (e.g., build-essential, libpq-dev, gcc) required by some Python packages; `uv sync` may fail when building wheels.
- Evidence:
  - `backend/Dockerfile` (builder stage) excerpts:

    RUN pip install --no-cache-dir uv
    COPY pyproject.toml uv.lock ./
    RUN uv sync --frozen --no-install-project

    (no `apt-get update` / `apt-get install` for build tools present)

  - Packages in `pyproject.toml` include `psycopg[binary]`, `uvicorn[standard]`, `sqlmodel` — depending on availability of prebuilt wheels, building may require system libs.

- Recommendation: Update the builder stage to install necessary OS packages before running `uv sync`. A minimal set to consider:
  - `apt-get update && apt-get install -y build-essential gcc libpq-dev curl ca-certificates` (adjust to the project's needs). Optionally remove them in a later layer to keep the runtime image small.
  - Alternatively, pre-generate wheels or use a base image that includes build tooling for deterministic builds.

- Rule candidate: When syncing Python dependencies in a slim Docker image, ensure required OS-level build dependencies are installed or use a base image that provides them.

### Finding 4
- Severity: medium
- Category: workflow / simplification
- Summary: Frontend dependencies are not locked; no `package-lock.json` (or equivalent) is committed, making frontend Docker builds non-deterministic.
- Evidence:
  - `frontend/package.json` exists with caret (^) ranges in devDependencies (e.g., `"svelte": "^5.0.0"`, `"@sveltejs/kit": "^2.0.0"`), but there is no committed `package-lock.json` or `pnpm-lock.yaml` in the repo.

- Recommendation: Commit a package-manager lockfile for the frontend (e.g., `package-lock.json` generated by `npm ci`/`npm install --package-lock`) or adopt `pnpm`/`yarn` with their lockfile. Update the frontend Dockerfile to use `npm ci` (preferred for reproducible installations) instead of `npm install`.

- Rule candidate: When building frontend Docker images, commit and use a package-manager lockfile to ensure reproducible builds.
