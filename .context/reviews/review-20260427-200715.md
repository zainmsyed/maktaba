## Findings

### Finding 1
- Severity: medium
- Category: regression / integration
- Summary: The frontend initially continued to show the library's built-in comment editor instead of the app's persistent note editor, causing notes typed into the popup to not be persisted or appear in the sidebar.
- Evidence:
  - Reported behaviour during development: console showing the library's comment payload ("Editing highlight ... {comment: '...'}") and notes disappearing on refresh (user report).
  - Code changes addressing this are in `frontend/src/routes/library/[documentId]/+page.svelte` (introduced `editHighlightPopup`, `noteEditor` snippet and passed it into PdfHighlighter) and `frontend/tests/mocks/MockPdfHighlighter.svelte` (added pinned/edit popup simulation). See `editHighlightPopup` and `noteEditor` snippets in `+page.svelte` and the pinned logic in the mock.
  - Prior to the fix, the Vitest run failed (unable to find textarea / timeout). After adapting the popup + mock and test wait logic the full frontend suite passes locally.
- Recommendation: Keep the custom `editHighlightPopup` approach and ensure the PdfHighlighter prop contract is covered by the MockPdfHighlighter in tests. Consider adding an end-to-end smoke test that exercises the popup → save → sidebar round-trip to catch regressions in future refactors.
- Rule candidate: When replacing or wrapping a third-party component's UI hooks, update the test mocks to mirror the new prop/behavior and add a small E2E smoke test for the integration.

### Finding 2
- Severity: low
- Category: dead-code / nit
- Summary: Duplicate import present in `backend/app/main.py`.
- Evidence: `from pathlib import Path` appears twice (lines ~4 and ~16 in `backend/app/main.py`). This is a cosmetic issue and should be removed to keep the module clean and to satisfy linters.

### Finding 3
- Severity: low
- Category: feature-gap / simplification
- Summary: Note CRUD implemented includes create (POST) and update (PATCH) but there is no DELETE `/api/notes/{note_id}` server endpoint or a frontend affordance to delete notes.
- Evidence: `backend/app/main.py` exposes `POST /api/documents/{document_id}/notes` and `PATCH /api/notes/{note_id}` but no `DELETE /api/notes/{note_id}`. Frontend `highlight-api.ts` provides create/update but no deleteNote helper.
- Recommendation: If note deletion is desired by product, add `DELETE /api/notes/{note_id}` and a small frontend client method (and UI affordance), plus tests for deletion and for deleting notes attached to highlights (and the cleanup semantics). If deletion is intentionally out of scope, document that in Story 008 so product/QA know the behaviour is deliberate.
- Rule candidate: —

### Finding 4
- Severity: low
- Category: test-gap / flakiness
- Summary: Tests that rely on fake timers + DOM updates can be fragile. The reader test initially timed out until the mock was updated and `waitFor` was used. Using fake timers with complex DOM/tick interactions may cause intermittent timing issues.
- Evidence: `frontend/tests/reader-page.test.ts` initially failed finding the note textarea and later timed out; the test was adjusted (added `waitFor` and updated MockPdfHighlighter) to stabilize it. The mock now supports pinned edit popups.
- Recommendation: Prefer using real timers for DOM-driven tests or add a test helper that advances timers and flushes microtasks consistently. Add comments in the affected tests explaining why fake timers are used and how to debug timeout problems (or prefer `vi.useRealTimers()` where possible).
- Rule candidate: Add a test-pattern note in the repo (`frontend/tests/README.md`) describing stable approaches for DOM + timer-driven tests (useRealTimers vs explicit waitFor + advanceTimers).

---

### Finding 5
- Severity: low
- Category: simplification
- Summary: `openHighlightNoteEditor` falls back to the last highlight if the requested highlight id is not found. This silent fallback can hide upstream logic errors (wrong id passed) and may lead to surprising UX.
- Evidence: `openHighlightNoteEditor` in `frontend/src/routes/library/[documentId]/+page.svelte` uses `getHighlightById(highlightId) ?? [...highlights].slice(-1)[0] ?? null`.
- Recommendation: Replace silent fallback with a clearer behaviour: either bail with a console.warn and no-op, or surface an explicit error. If fallback is intentionally helpful (e.g., newly created highlight not yet in `highlights` array), add a short comment and a small comment in code explaining the intent.
- Rule candidate: —
