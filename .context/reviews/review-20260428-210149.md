### Finding 2
- Severity: medium
- Category: dead-code
- Summary: The reader page retains leftover inline-editor handlers and DOM refs that are no longer used after extracting NoteEditor.
- Evidence: frontend/src/routes/library/[documentId]/+page.svelte defines function handleNoteInput and variables noteAutosaveTimer and noteTextareaEl, but there are no references or bindings to that function/textarea in the current template. Example snippet:
  - function handleNoteInput(event: Event) { /* schedules noteAutosaveTimer */ }
  - let noteTextareaEl: HTMLTextAreaElement | null = null;
  Search shows handleNoteInput is never invoked.
- Recommendation: Remove unused code (handleNoteInput, noteAutosaveTimer, noteTextareaEl) or rewire them if the fallback inline editor is intentionally kept. Removing these avoids confusion and reduces maintenance burden.
- Rule candidate: When extracting a UI component, remove the parent's legacy handlers and element refs that were used only by the inline implementation.

### Finding 3
- Severity: medium
- Category: simplification
- Summary: Autosave and saved-state behavior is implemented in both NoteEditor (component) and the reader page (parent). This duplication increases maintenance burden and risks inconsistent UX.
- Evidence: frontend/src/components/NoteEditor.svelte contains autosave timer, runSave, saved state and spin UI. The parent (frontend/src/routes/library/[documentId]/+page.svelte) also manages noteSavedDraft, scheduleSavedNoteFade, setNoteStatus, and persistNoteDraft. Both layers touch save lifecycle.
- Recommendation: Consolidate autosave and immediate save-status UI inside the NoteEditor component and have the parent only react to saved/updated events. Keep persistNoteDraft as the single backend persistence surface invoked via NoteEditor.onSave; remove parent-side autosave timers and duplicate status variables where unused.
- Rule candidate: Prefer single-source-of-truth for autosave and save-status: implement in the component owning the input and expose events/async save API to parents.

### Finding 4
- Severity: low
- Category: workflow
- Summary: AGENTS.md is a placeholder and lacks basic project metadata (name, goal, stack, important paths) which makes onboarding and agent guidance brittle.
- Evidence: AGENTS.md contains empty placeholders for project name/goal/stack and fragile areas.
- Recommendation: Populate AGENTS.md with at least project name, goal, stack, critical paths, and the handful of fragile areas already noted (pgvector/bootstrap, OCR, highlight→viewer sync, embeddings pipeline). This improves handoffs for future stories and automated agents.
- Rule candidate: —

### Finding 5
- Severity: low
- Category: bug (error-handling)
- Summary: Some endpoints raise HTTP 500 with the raw exception string which can leak internal details and produce noisy client messages.
- Evidence: backend/app/main.py create_highlight wraps extraction in try/except and on exception raises HTTPException(status_code=500, detail=str(exc)). Similar patterns appear in other handlers.
- Recommendation: Map expected input/validation errors to 4xx responses (400/422) and log internal exceptions server-side while returning a generic 5xx message to clients. This reduces information leakage and makes client behavior easier to reason about in production.
- Rule candidate: Consider a rule: "Do not return raw exception messages in API HTTP 5xx responses; log internals and return a generic message."
