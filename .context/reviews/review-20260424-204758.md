## Findings

### Finding 1
- Severity: medium
- Category: bug
- Summary: The schema does not encode the delete-cascade behavior required by the PRD, so future hard deletes or cleanup jobs will leave orphan rows instead of deleting pages, highlights, notes, and jobs with their parent document.
- Evidence:
  - `backend/app/models.py` declares plain foreign keys such as `foreign_key="documents.id"` and `foreign_key="highlights.id"` without an explicit delete policy on `Page`, `Highlight`, `Note`, or `Job`.
  - SQLModel metadata inspection shows the relevant FKs all have `ondelete=None`:
    - `pages.document_id -> documents.id ondelete=None`
    - `highlights.document_id -> documents.id ondelete=None`
    - `notes.document_id -> documents.id ondelete=None`
    - `notes.highlight_id -> highlights.id ondelete=None`
    - `jobs.document_id -> documents.id ondelete=None`

- Recommendation: Add the intended `ondelete` policy to the foreign keys that should cascade or null out on parent deletion, then re-run the Postgres schema verification so the runtime database matches the PRD.

- Rule candidate: When the data model depends on delete cascades, declare the delete policy in the foreign key definition rather than relying on application-side cleanup.

### Finding 2
- Severity: low
- Category: test-gap
- Summary: Story 002 was verified manually, but there is still no automated regression test that proves the tables, pgvector extension, and indexes are created correctly after startup.

  - Verification for this story relied on manual compose startup and ad-hoc inspection rather than a repeatable automated test.

- Recommendation: Add a small backend integration test that boots the database or uses the compose stack, then asserts the expected tables, extension, and indexes exist. Hook it into CI so future schema changes fail fast.

- Rule candidate: When adding schema/bootstrap logic, include at least one automated Postgres smoke test for tables, extensions, and indexes.
