Findings

### 1) Workflow — Binary PDF file tracked in git
- Severity: medium
- Category: workflow
- Summary: A user-uploaded PDF (`data/pdfs/780668c60178a66e31394f81ad09256d6a9dca31b7578eac87477e9e3fcddc0b.pdf`) was staged/added in the working tree. Binary uploads in git history bloat the repo and can leak user data. The `.gitignore` did not exclude `data/pdfs/`.
- Evidence: `git status --porcelain` showed `A data/pdfs/780668c60178...pdf`. The `.gitignore` file lacked a `data/pdfs/` or `*.pdf` exclusion for the data directory.
- Recommendation: Remove the PDF from git history (`git rm --cached`), add `data/pdfs/` and `data/epubs/` to `.gitignore`, and ensure `data/` retains only `.gitkeep` for directory structure. If a test fixture PDF is needed, keep it under `tests/fixtures` with a documented purpose or use Git LFS.
- Rule candidate: —

### 2) Bug / Regression — DEPLOY.md references merged `scaffold` branch
- Severity: low
- Category: regression
- Summary: The deployment guide instructed users to `git pull origin scaffold`, but the `scaffold` branch was fast-forward merged into `main`. New users following this would pull an outdated branch reference.
- Evidence: `DEPLOY.md` contained `git pull origin scaffold` in three places (Quick Start, prod Updates, local Updates). Git history shows `scaffold` was merged into `main` via fast-forward.
- Recommendation: Update all occurrences to `git pull origin main`.
- Rule candidate: —

### 3) Test gap — No frontend tests for delete button right-alignment fixes
- Severity: medium
- Category: test-gap
- Summary: Two recent CSS fixes added `margin-left: auto` to `.delete-btn` (library cards) and `.paper-note-del` (sidebar note rows) to keep delete buttons right-aligned when sibling content is absent. Neither fix had a regression test.
- Evidence: `frontend/src/routes/library/+page.svelte` and `frontend/src/routes/library/[documentId]/+page.svelte` contained the fixes. The frontend test suite had library-page and reader-page tests but none asserting delete button alignment when `highlight_count`/`note_count` are zero or note content is empty.
- Recommendation: Add Vitest + JSDOM tests that:
  1. Render a library card with zero highlight/note counts and assert the `.delete-btn` element is right-aligned.
  2. Render a sidebar annotation card with no note text and assert the `.paper-note-del` element is right-aligned.
- Rule candidate: —

### 4) Test gap — No frontend test for popup note backend highlight ID resolution
- Severity: medium
- Category: test-gap
- Summary: The `backendHighlightIdByStoreId` Map is critical for popup note saves: it translates temporary client-side highlighter IDs to persisted backend UUIDs. A regression here would cause popup notes to silently fail or target the wrong highlight. No test exercised this mapping.
- Evidence: `frontend/src/routes/library/[documentId]/+page.svelte` defines `backendHighlightIdByStoreId` and uses it in `resolveBackendHighlightId`, `savePopupHighlightNote`, and `updateHighlightColor`. The reader-page tests exercised sidebar notes and popup UI but did not verify that a popup note save sends the backend UUID rather than the temporary store ID.
- Recommendation: Extend the reader-page test suite to simulate a highlight creation (which populates `backendHighlightIdByStoreId`), then trigger a popup note save and assert the `POST /api/documents/{id}/notes` request body contains `highlight_id` set to the backend UUID, not the temporary ID.
- Rule candidate: —

### 5) Dead code — `popup-viewport-guard.ts` exports unused internal helpers
- Severity: low
- Category: dead-code
- Summary: `fallow` flagged 6 unused exports in `frontend/src/lib/popup-viewport-guard.ts`: `readBaseTop`, `clampTop`, `applyTop`, `guardPopupContainer`, `guardPopupInView`, and `PopupViewportGuardOptions`. Only `popupViewportGuard` and `keepPopupInView` were imported by consumers.
- Evidence: `fallow --root frontend --only dead-code` reported 5 unused exports + 1 unused type export from `popup-viewport-guard.ts`. `grep` confirmed only `popupViewportGuard` (reader) and `keepPopupInView` (NotePopup) are imported externally.
- Recommendation: Remove `export` from functions/types that are only used internally within the module. Keep them as module-private helpers.
- Rule candidate: —

### 6) Info — Duplicate `margin-left: auto` pattern could be a utility
- Severity: info
- Category: simplification
- Summary: The same layout fix (`margin-left: auto` on an action button inside a flex row) was applied independently in two components. Without a shared utility, future components may repeat the same fragile `justify-content: space-between` pattern.
- Evidence: `.delete-btn` in `frontend/src/routes/library/+page.svelte` and `.paper-note-del` in `frontend/src/routes/library/[documentId]/+page.svelte` both needed the same fix.
- Recommendation: Add a small utility class (e.g., `.push-right { margin-left: auto; }`) in `frontend/src/lib/theme.css` or a component-level helper, and apply it consistently.
- Rule candidate: Add small layout utility classes for common flex patterns.

### 7) Info — Dead CSS selectors remain in reader page
- Severity: info
- Category: dead-code
- Summary: Manual inspection found several CSS selectors in the reader's `<svelte:head>` block with no matching markup: `.reader-stage .reader-topbar-actions a`, `.reader-highlights-panel .grid.grid-cols-2 button`, `.reader-navigation-panel .flex.items-center.gap-2 > button`, `.reader-sidebar > section.reader-{highlights,status,document,navigation,zoom}-panel`, and `.reader-sidebar .note-item`.
- Evidence: A grep-based pass over the markup (post-`</svelte:head>`) found zero occurrences of these class names in HTML/Svelte markup.
- Recommendation: Remove the orphaned selectors to reduce CSS payload and improve maintainability.
- Rule candidate: —
