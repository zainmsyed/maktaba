### 1) Workflow — Binary PDF file tracked in git
- Severity: medium
- Category: workflow
- Summary: A user-uploaded PDF (`data/pdfs/780668c60178a66e31394f81ad09256d6a9dca31b7578eac87477e9e3fcddc0b.pdf`) is staged/added in the working tree. Binary uploads in git history bloat the repo and can leak user data. The `.gitignore` does not exclude `data/pdfs/`.
- Evidence: `git status --porcelain` shows `A data/pdfs/780668c60178a66e31394f81ad09256d6a9dca31b7578eac87477e9e3fcddc0b.pdf`. The `.gitignore` file lacks a `data/pdfs/` or `*.pdf` exclusion for the data directory.
- Recommendation: Remove the PDF from git history (`git rm --cached`), add `data/pdfs/` and `data/epubs/` to `.gitignore`, and ensure `data/` retains only `.gitkeep` for directory structure. If a test fixture PDF is needed, keep it under `tests/fixtures` with a documented purpose or use Git LFS.
- Rule candidate: —

### 2) Bug / Regression — DEPLOY.md references merged `scaffold` branch
- Severity: low
- Category: regression
- Summary: The deployment guide instructs users to `git pull origin scaffold`, but the `scaffold` branch was fast-forward merged into `main`. New users following this will pull an outdated branch reference (or fail if the branch is deleted).
- Evidence: `DEPLOY.md` line `git pull origin scaffold` appears twice (Quick Start and Updates sections). The git history shows `scaffold` was merged into `main` via fast-forward.
- Recommendation: Update both occurrences to `git pull origin main` (or `git pull origin bug-fixes-scanned-pdfs` if deploying from a feature branch is intentional, but `main` is the correct long-term default).
- Rule candidate: —

### 3) Test gap — No frontend tests for delete button right-alignment fixes
- Severity: medium
- Category: test-gap
- Summary: Two recent CSS fixes added `margin-left: auto` to `.delete-btn` (library cards) and `.paper-note-del` (sidebar note rows) to keep delete buttons right-aligned when sibling content is absent. Neither fix has a regression test.
- Evidence: `frontend/src/routes/library/+page.svelte` and `frontend/src/routes/library/[documentId]/+page.svelte` contain the fixes. The frontend test suite (`frontend/tests/`) has library-page and reader-page tests but none assert computed styles or DOM layout for delete button alignment when `highlight_count`/`note_count` are zero or note content is empty.
- Recommendation: Add Vitest + JSDOM tests that:
  1. Render a library card with zero highlight/note counts and assert the `.delete-btn` element has `margin-left: auto` (or is the rightmost child in its flex container).
  2. Render a sidebar annotation card with no note text and assert the `.paper-note-del` element is right-aligned.
- Rule candidate: —

### 4) Test gap — No frontend test for popup note backend highlight ID resolution
- Severity: medium
- Category: test-gap
- Summary: The `backendHighlightIdByStoreId` Map is critical for popup note saves: it translates temporary client-side highlighter IDs to persisted backend UUIDs. A regression here would cause popup notes to silently fail or target the wrong highlight. No test exercises this mapping.
- Evidence: `frontend/src/routes/library/[documentId]/+page.svelte` defines `backendHighlightIdByStoreId` and uses it in `resolveBackendHighlightId`, `savePopupHighlightNote`, and `updateHighlightColor`. The reader-page tests exercise sidebar notes and popup UI but do not verify that a popup note save sends the backend UUID rather than the temporary store ID.
- Recommendation: Extend the reader-page test suite to simulate a highlight creation (which populates `backendHighlightIdByStoreId`), then trigger a popup note save and assert the `POST /api/documents/{id}/notes` request body contains `highlight_id` set to the backend UUID, not the temporary ID.
- Rule candidate: —

### 5) Info — Potential dead CSS selectors remain unchecked
- Severity: info
- Category: dead-code
- Summary: The unused `.paper-hl-del-btn` selector was removed in commit cf8ba88, but the repository has not had a systematic pass to find other orphaned CSS rules. The reader `<svelte:head>` style block is large (~1000 lines) and may contain additional unused selectors.
- Evidence: Manual inspection of `frontend/src/routes/library/[documentId]/+page.svelte` shows many override selectors (e.g., `.reader-sidebar .bg-slate-950\/70`) that may be stale if the underlying component markup changed. No automated CSS purge or usage check is configured.
- Recommendation: Run a one-time grep pass (or introduce a lightweight CSS analyzer in CI) to find class names defined in `<style>` blocks that do not appear in the corresponding markup. Consider consolidating repeated flex patterns into documented utility classes.
- Rule candidate: —

### 6) Info — Duplicate `margin-left: auto` pattern could be a utility
- Severity: info
- Category: simplification
- Summary: The same layout fix (`margin-left: auto` on an action button inside a flex row) was applied independently in two components. Without a shared utility, future components may repeat the same fragile `justify-content: space-between` pattern.
- Evidence: `.delete-btn` in `frontend/src/routes/library/+page.svelte` and `.paper-note-del` in `frontend/src/routes/library/[documentId]/+page.svelte` both needed the same fix.
- Recommendation: Consider adding a small utility class (e.g., `.push-right { margin-left: auto; }`) in `frontend/src/lib/theme.css` or a component-level helper, and apply it consistently. This makes intent explicit and prevents regressions when flex layouts change.
- Rule candidate: Add small layout utility classes for common flex patterns.
