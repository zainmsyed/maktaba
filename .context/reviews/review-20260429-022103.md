### Finding 2
- Severity: medium
- Category: simplification
- Summary: Autosave/save-status behavior is implemented both inside NoteEditor.svelte and in the reader page, causing duplication and a risk of inconsistent UX/state.
- Evidence: NoteEditor.svelte implements its own autosave timer, save(), status states, and saved-fade logic (autosaveTimer, savedClearTimer, runSave, scheduleSavedNoteFade). The reader page also manages noteDraft, noteSavedDraft, noteSavedClearTimer, setNoteStatus, scheduleSavedNoteFade, and persistNoteDraft which performs the backend save. Example excerpts:
  - NoteEditor.svelte handles input and autosave:

    ```js
    let autosaveTimer: number | null = null;
    function handleInput(e: Event) {
      draft = target.value;
      autosaveTimer = window.setTimeout(() => { autosaveTimer = null; void runSave(); }, 500);
      onChange?.(draft);
    }
    export function save() { return runSave(); }
    ```

  - Reader page also schedules saves and status updates:

    ```js
    function handleNoteInput(event: Event) { /* schedules noteAutosaveTimer */ }
    async function persistNoteDraft(target, note, draft, revision) { /* create/update via API and set saved state */ }
    ```

- Recommendation: Consolidate autosave and save-status UI into the NoteEditor component (it already exposes onSave/onChange and a save() API). The reader page should treat NoteEditor as the single source of truth for input timers/status and only handle high-level persistence events (onSave, saved) and server-side merging (upsertNote). This removes duplicate timers and minimizes subtle race conditions between component and parent.
- Rule candidate: Prefer single-source-of-truth for autosave and immediate save-status UI: the component owning the input should manage timers and UX, exposing an async save API and high-level events to parents.

### Finding 5
- Severity: low
- Category: bug (error-handling)
- Summary: Several handlers convert arbitrary exceptions into HTTP 500 responses returning the raw exception string as the detail, which risks leaking internal implementation details to API consumers.
- Evidence: create_highlight wraps the PyMuPDF extraction in a try/except and returns the exception string in the 500 response:

    ```py
    except Exception as exc:
        raise HTTPException(status_code=status.HTTP_500_INTERNAL_SERVER_ERROR, detail=str(exc))
    ```

  Similar patterns may exist in other endpoints that catch Exception and embed str(exc) in the response.

- Recommendation: Replace raw-exception responses with: (a) logging the full exception server-side (including traceback), and (b) returning a generic 5xx message to the client (e.g., "Internal server error during PDF extraction"). For expected failure modes (validation, missing file, unsupported geometry), map to explicit 4xx/503 statuses with clear messages. This reduces information leakage and improves security/observability.
- Rule candidate: Do not return raw exception strings in HTTP 5xx responses; log details internally and return a generic message to clients.
