## Findings

### Finding 1
- Severity: medium
- Category: bug
- Summary: 1-page document shows 0% progress in library but 100% in reader.
- Evidence: In `frontend/src/routes/library/+page.svelte`, `readingProgressPercent` returns 0 for `reachedPage <= 1` before the `total === 1` branch can execute, so a single-page document always reads 0%. The reader topbar uses `Math.max(1, Math.round((currentPage / totalPages) * 100))`, which returns 100% for the same document.
- Recommendation: Align the two formulas. Change the library baseline check to `if (reachedPage < 1) return 0;` so a 1-page document at page 1 returns 100%, or extract a shared `computeProgressPercent(currentPage, totalPages)` helper used by both views.
- Rule candidate: When progress logic is duplicated across views, extract a single tested helper to prevent drift.

### Finding 2
- Severity: medium
- Category: bug
- Summary: NoteEditor `runSave` can overwrite user input while a save is in flight.
- Evidence: `frontend/src/components/NoteEditor.svelte` lines 63-82. `runSave` captures `draft` at call time, awaits `onSave(draft)`, then unconditionally assigns `draft = newContent`. If the user types during the network request, those keystrokes are clobbered when the promise resolves.
- Recommendation: Snapshot `draft` before the `await` and only update `draft` after save if it has not changed in the interim (e.g., compare the pre-save snapshot to the current `draft`).
- Rule candidate: —
